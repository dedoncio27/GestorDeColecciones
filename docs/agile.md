# Metodologías Ágiles: Desarrollo de Software Moderno

Este documento resume la investigación sobre metodologías ágiles aplicada al desarrollo de software, centrada en la flexibilidad, la entrega continua y la colaboración.

## 1. ¿Qué es Agile?

Agile (o desarrollo ágil) no es una "receta" única, sino un **conjunto de valores y principios** que priorizan a las personas y la adaptación al cambio sobre los procesos rígidos.

*   **Objetivo**: Entregar software funcional de forma rápida y frecuente, permitiendo que el cliente vea el progreso y pueda pedir cambios sin que eso signifique tirar meses de trabajo a la basura.

## 2. Metodología Scrum

Scrum es un marco de trabajo ágil diseñado para equipos que trabajan en proyectos complejos. Se basa en ciclos de trabajo cortos y reuniones constantes.

### Conceptos principales:
*   **Sprints**: Ciclos de trabajo fijos (normalmente de 2 a 4 semanas) donde se desarrolla una parte específica del producto.
*   **Product Backlog**: La lista maestra de todo lo que la aplicación necesita tener (funcionalidades, correcciones, mejoras).
*   **Sprint Backlog**: Las tareas seleccionadas del Product Backlog para hacerse en el sprint actual.
*   **Roles**:
    *   *Product Owner*: Define qué se hace (prioriza el valor).
    *   *Scrum Master*: Ayuda al equipo a seguir el proceso y elimina obstáculos.
    *   *Equipo de Desarrollo*: Los que pican el código.
*   **Reviews**: Reuniones al final del sprint para mostrar lo que se ha construido.

## 3. Metodología Kanban

Kanban es una metodología visual que se centra en el **flujo continuo de trabajo** y en no sobrecargar al equipo.

*   **El Tablero**: Se usa un tablero con columnas (To Do, Doing, Done).
*   **WIP (Work In Progress)**: Se limita cuántas tareas pueden estar en la columna "Haciendo" a la vez para evitar cuellos de botella.
*   **Uso**: A diferencia de Scrum, no hay ciclos fijos (sprints). El trabajo fluye constantemente según se van terminando tareas.

## 4. Diferencias: Scrum vs Kanban

| Característica | Scrum | Kanban |
| :--- | :--- | :--- |
| **Cadencia** | Sprints fijos (ej. 2 semanas) | Flujo continuo |
| **Roles** | Roles definidos obligatorios | No requiere roles específicos |
| **Cambios** | No se permiten cambios durante el sprint | Se pueden añadir tareas en cualquier momento |
| **Métrica** | Velocidad (puntos por sprint) | Tiempo de ciclo (cuánto tarda una tarea) |

## 5. ¿Cuándo usar cada una?

*   **Usa Scrum cuando**: Estás construyendo un producto desde cero, necesitas mucha estructura y el equipo se beneficia de tener metas claras cada dos semanas.
*   **Usa Kanban cuando**: Tu proyecto ya está en marcha y necesitas gestionar tareas de mantenimiento, soporte o mejoras constantes donde las prioridades cambian día a día.
